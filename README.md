# Лабораторная работа №3. Очередь задач: итераторы и генераторы

## Описание

Данный проект реализует подсистему приёма задач в абстрактной платформе обработки задач. Задачи могут поступать из различных источников, не связанных наследованием, но обязанных реализовывать единый поведенческий контракт, описанный через `typing.Protocol`.

Цель лабораторной работы —   Научиться реализовывать пользовательские коллекции и ленивую обработку задач.

Под задачей понимается единица работы, содержащая:
- `id` — уникальный идентификатор;
- `description` — текстовое описание задачи;
- `priority` — приоритет задачи (целое число);
- `status` — статус задачи (строка).
- `_created_at` — дата и время создания задачи.

Задача может поступать из различных источников, таких как файлы, API-заглушки или генераторы, и подлежит дальнейшей обработке системой.

## Функциональность

Проект реализует следующие функции:

- **Определение контракта через Protocol**: 
  Определён протокол `Source` с методом `get_tasks() -> list[Task]`, который обязывает все источники реализовывать единый интерфейс.

- **Источники задач**:
  - `ApiSource`: имитирует получение задач из внешнего API через заглушку с предопределёнными данными.
  - `FileSource`: читает задачи из текстового файла в формате `id:description:priority:status`, игнорируя пустые строки и логируя ошибки парсинга.
  - `TasksGenerator`: генерирует случайное количество случайных задач с идентификаторами.

- **Сбор задач**:
  - Класс `TaskTaker` принимает список источников и проверяет их соответствие протоколу `Source` с помощью `isinstance()`.
  - Автоматически пропускает объекты, не соответствующие контракту, с логированием ошибки.
  - Реализована проверка возвращаемого значения: `get_tasks()` должен возвращать список объектов `Task`, иначе источник пропускается.

- **Обработка ошибок**:
  - При ошибках чтения файла (`FileNotFoundError`) или парсинга строк — ошибка логируется, обработка продолжается.
  - При исключениях в методе `get_tasks()` любого источника — ошибка логируется, остальные источники продолжают обрабатываться.

- **Логирование**:
  - Используется встроенный модуль `logging` для отслеживания процесса сбора задач.
  - Логируются как успешные операции, так и все ошибки.

- **Task Model**:
  - Класс `Task` реализует атрибуты `id`, `description`, `priority`, `status` и защищённый атрибут `_created_at`.
  - Атрибут `_created_at` доступен только для чтения через свойство `created_at`, что обеспечивает защиту инварианта модели.

- **Tasks Queue**: 
  - Класс `TaskQueue` реализует пользовательскую коллекцию для хранения и управления задачами.
  - Класс `TaskFilter` обеспечивает ленивую фильтрацию задач с использованием генераторов.
  - Класс `TaskIterator` реализует явный выброс исключения StopIteration при исчерпании элементов
