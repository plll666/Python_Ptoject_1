# Платформа обработки задач

Курсовой проект, реализующий абстрактную платформу для приёма, хранения и обработки задач. Проект выполнен в рамках 4 лабораторных работ.

## Описание предметной области

Платформа предназначена для:
- приёма задач из различных источников;
- хранения и организации задач;
- последовательной и асинхронной обработки задач;
- расширения функциональности без изменения существующего кода.

**Задача** — единица работы, содержащая:
- `id` — уникальный идентификатор;
- `description` — текстовое описание;
- `priority` — приоритет (1-5);
- `status` — статус (`new`, `in_progress`, `completed`, `deleted`);
- `_created_at` — время создания (только чтение).

## Лабораторная работа №1. Источники задач и контракты

**Цель:** Освоить duck typing и контрактное программирование.

### Реализация

| Компонент | Описание |
|-----------|----------|
| `Source` Protocol | `@runtime_checkable` протокол с методом `get_tasks() -> list[Task]` |
| `TaskHandler` Protocol | Протокол асинхронного обработчика `handle_task(task)` |
| `isinstance()` проверка | Runtime-проверка соответствия источников контракту |

### Источники задач

- **`ApiSource`** — имитирует получение задач из внешнего API;
- **`FileSource`** — читает задачи из файла формата `id:description:priority:status`;
- **`TasksGenerator`** — генерирует случайные задачи.

### Ключевые особенности

- Источники не связаны наследованием, реализуют только поведенческий контракт;
- `TaskTaker` собирает задачи из списка источников с валидацией;
- Ошибки парсинга и чтения логируются, но не прерывают обработку.

## Лабораторная работа №2. Модель задачи: дескрипторы и @property

**Цель:** Управление доступом к атрибутам и защита инвариантов доменной модели.

### Реализация

#### Data-дескрипторы (с валидацией при записи)

| Дескриптор | Валидация | Особенность |
|------------|-----------|-------------|
| `IdDescriptor` | Непустая строка | Неизменяем после инициализации |
| `PriorityDescriptor` | Число 1-5 | Всегда валидируется |
| `StatusDescriptor` | Один из 4 статусов | Константа `VALID_STATUSES` |

#### Non-data дескриптор

- **`DescriptionDescriptor`** — простой геттер без валидации.

#### @property (вычисляемые свойства)

- **`is_active`** — задача активна (не завершена и не удалена);
- **`is_high_priority`** — высокий приоритет (4-5);
- **`created_at`** — время создания (только чтение через `_created_at`).

### Обработка исключений

- `ValueError` при нарушении валидации;
- `AttributeError` при попытке изменить `id` после создания;
- `IndexError` при обращении к несуществующему индексу.

## Лабораторная работа №3. Очередь задач: итераторы и генераторы

**Цель:** Реализация пользовательских коллекций и ленивая обработка задач.

### Реализация

| Компонент | Описание |
|-----------|----------|
| `TaskQueue` | Коллекция с поддержкой итерации, индексации `__getitem__` и `__len__` |
| `TaskIterator` | Явный итератор с управлением состоянием и `StopIteration` |
| `TaskFilter` | Ленивая фильтрация через генератор, поддержка цепочек `filter()` |

### Ключевые особенности

- Ленивая фильтрация не загружает данные в память;
- Повторный обход очереди возможен через создание нового итератора;
- Методы `filter_by_status()` и `filter_by_priority()`;
- Совместимость со стандартными конструкциями Python (`for`, `list`, `sum`).

## Лабораторная работа №4. Асинхронный исполнитель задач

**Цель:** Реализация приложений с асинхронной моделью управления.

### Реализация

| Компонент | Описание |
|-----------|----------|
| `AsyncExecutor` | Асинхронный исполнитель с контекстным менеджером |
| `AsyncTaskQueue` | Приоритетная асинхронная очередь |
| `TaskHandler` Protocol | Контракт для обработчиков задач |

### Обработчики задач

- **`PrintHandler`** — печать задач;
- **`LoggingHandler`** — логирование задач;
- **`StatsHandler`** — сбор статистики (счётчик обработанных, высокоприоритетных);
- **`RetryHandler`** — обработка с повторными попытками;
- **`FilterHandler`** — фильтрация по минимальному приоритету.

### Ключевые особенности

- `__aenter__`/`__aexit__` для управления ресурсами;
- Worker'ы завершаются сигналом `None` в очереди;
- Централизованное логирование и обработка ошибок;
- Расширяемая архитектура: новые обработчики регистрируются через `register_handler()`.


## Технологии и подходы

| Технология | Применение |
|------------|------------|
| `typing.Protocol` | Контракты источников и обработчиков |
| `@runtime_checkable` | Runtime-проверка duck typing |
| Data descriptors | Валидация при записи атрибутов |
| Non-data descriptors | Простое управление доступом |
| `@property` | Вычисляемые свойства |
| `__iter__` / `__next__` | Протокол итерации |
| Генераторы | Ленивая фильтрация |
| `async` / `await` | Асинхронная обработка |
| Контекстные менеджеры | Управление ресурсами |
